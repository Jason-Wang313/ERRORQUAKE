# Datasheet for ERRORQUAKE-10K

Following Gebru et al. (2021), "Datasheets for Datasets."

## Motivation

**For what purpose was the dataset created?**
ERRORQUAKE-10K was created to study error severity distributions in open-weight LLMs. Standard hallucination benchmarks report a single error rate; ERRORQUAKE provides a continuous 9-level severity scale that enables fitting per-model severity distributions and computing the Gutenberg-Richter severity distribution index (b-value).

**Who created the dataset and on behalf of which entity?**
Anonymous (double-blind submission to NeurIPS 2026 E&D track).

**Who funded the creation of the dataset?**
No external funding. All inference was performed using NVIDIA NIM API free-tier credits.

## Composition

**What do the instances that comprise the dataset represent?**
Each instance is a (query, model_response, severity_score) triple. Queries are factual questions across 8 domains (BIO, LAW, HIST, GEO, SCI, TECH, FIN, CULT) and 5 difficulty tiers.

**How many instances are there in total?**
10,000 unique queries x 21 models = 210,000 evaluation instances. Each instance has a primary and secondary judge score plus a resolved final score.

**Does the dataset contain all possible instances or is it a sample?**
It is a stratified sample: 250 queries per (domain x tier) cell, for 8 domains x 5 tiers = 40 cells.

**What data does each instance consist of?**
- `query_id`: Unique identifier (e.g., BIO_T1_0036)
- `question`: The factual question
- `ground_truth`: Reference answer
- `domain`: One of 8 domains
- `tier`: Difficulty level (1-5)
- `model_name`: Which model generated the response
- `response_text`: The model's full response
- `primary_score`: Score from primary judge (0.0-4.0)
- `secondary_score`: Score from secondary judge
- `final_score`: Resolved severity score
- `resolution_method`: How primary/secondary were combined

**Is any information missing from individual instances?**
Some instances have missing secondary scores (judge call failed). These are flagged with `resolution_method: "primary_only"`.

**Are there recommended data splits?**
No predefined splits. For cross-validation, we recommend cross-domain jackknife (leave one domain out).

## Collection Process

**How was the data associated with each instance acquired?**
Queries were generated by a frontier LLM (DeepSeek-V3.2 via NIM) prompted to produce stratified factual questions. Approximately 6% of queries were regenerated after a tier-calibration audit. Model responses were collected by sending each query to each model via NVIDIA NIM API at temperature=0.

**What mechanisms or procedures were used to collect the data?**
Automated pipeline: query generation -> model evaluation -> dual-judge scoring. All inference via NVIDIA NIM API.

**Who was involved in the data collection process?**
The first author designed the pipeline and ran all experiments. A single expert rater performed the 340-item overcall audit and 100-item human validation pilot.

**Over what timeframe was the data collected?**
March-April 2026.

## Preprocessing/Cleaning/Labeling

**Was any preprocessing/cleaning/labeling of the data done?**
Severity scores are quantized to 0.5 increments. Dual-judge resolution: if |primary - secondary| < 0.5, use primary; if < 1.5, average; if >= 1.5, tiebreak with third judge or flag for human review. A 340-item manual audit found 33.5% overcall rate at score 2.0.

## Uses

**What tasks has the dataset been used for?**
Fitting per-model severity distributions, computing b-values, testing matched-accuracy discrimination, scaling analysis, micro-error prediction.

**Is there anything about the composition of the dataset or the way it was collected and preprocessed/cleaned/labeled that might impact future uses?**
(1) All queries are LLM-generated, not human-authored. (2) Open-weight models only. (3) 33.5% overcall rate at score 2.0. (4) Judge ICC(2,k=2) = 0.545.

**Are there tasks for which the dataset should not be used?**
Should not be used as a leaderboard or for model ranking by accuracy alone. The severity distribution is the informative quantity, not the error rate.

## Distribution

**How will the dataset be distributed?**
Via HuggingFace (CC-BY-4.0 license for data).

**When will the dataset be first distributed?**
Upon paper acceptance or preprint posting.

## Maintenance

**Who will be maintaining the dataset?**
The first author. The benchmark will be maintained for 3 years with annual model additions as new open-weight models become available.

**Will the dataset be updated?**
Yes. New model evaluations will be added annually. The query set is fixed (for comparability).

**How can the owner/curator/manager of the dataset be contacted?**
Via the anonymous HuggingFace dataset page or the GitHub repository.
